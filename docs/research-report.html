Meeterz: recording meeting transcripts on macOS

Deep research into the right architecture for a Mac app that records Teams calls (system audio) and in-person meetings (microphone), transcribes locally with Whisper, and organizes everything with notes and folders.

Verdict

Electron is the right choice — and the hard problem just got easy

The biggest risk in this project — capturing macOS system audio without ugly hacks — was solved in 2025. As of Electron 39, Chromium uses Apple's CoreAudio Tap API by default for desktop audio capture on macOS. That means Teams call audio and your microphone can be captured as two independent streams, driver-free (no BlackHole/Soundflower), straight from Electron. Granola — the category-leading, Apple-quality meeting notes app — is itself an Electron app, which validates both the stack and the "premium UX in Electron" premise.

Verified findings

Electron 39 makes macOS system-audio capture native

High · 5 panels passed

Official Electron docs (fetched live): "As of Electron v39.0.0-beta.4, Chromium made Apple's new CoreAudio Tap API the default for desktop audio capture" on macOS. Before that (Electron 31–38), loopback needed Chromium feature-flag hacks via the electron-audio-loopback plugin; that plugin's author confirms Electron 39+ makes it unnecessary.

Electron desktopCapturer docselectron#47490

Core Audio process taps are the underlying OS capability

High · 4 panels passed

Apple's official API (macOS 14.2+, reliable from ~14.4): CATapDescription → AudioHardwareCreateProcessTap → aggregate device → IOProc. The tap behaves like a microphone input — no virtual audio driver. Guilherme Rambo's AudioCap is the open-source reference implementation.

Apple docsAudioCap

Dual-channel capture (mic + system, separate streams) is proven

High · 3 panels passed

The mic-speaker-streamer example app was code-inspected: it captures getUserMedia (mic) and getDisplayMedia loopback (system) as two distinct MediaStreams simultaneously, recording both into WAV — exactly Meeterz's requirement (Teams audio on one channel, room audio on the other).

mic-speaker-streamer

Granola validates Electron for a premium meeting-notes app

High · 2 panels passed

First-party source (Granola careers page): "Granola is an Electron app with deep OS integrations: system audio capture, accessibility APIs, webcam video streaming" — React/TypeScript/Node. Their capture runs in a bundled native Swift helper piping PCM to Electron; that remains the fallback plan if Electron 39's built-in path disappoints.

granola.ai

Whisper on Apple Silicon: use MLX or whisper.cpp+CoreML; never faster-whisper

Medium · 2 panels passed

M4 benchmark (large model): mlx-whisper 1.02s ≈ insanely-fast-whisper 1.13s ≈ whisper.cpp+CoreML 1.23s, while faster-whisper takes 6.96s because CTranslate2 is CPU-only on macOS. NVIDIA Parakeet TDT (via FluidAudio CoreML at 0.19s or parakeet-mlx at 0.50s) beats all Whisper builds 2–5× with equal-or-better English accuracy — worth adopting for English meetings. Single-machine hobbyist benchmark, but tiers independently corroborated.

Diarization (who-said-what) is available open-source, post-meeting

High · 2 panels passed

Argmax's open-source Swift SDK (MIT) ships WhisperKit (transcription, Core ML) and SpeakerKit (Pyannote v4 diarization, Core ML, macOS 13+). Only real-time diarized transcription is gated behind their commercial Pro SDK — irrelevant here since post-meeting processing was chosen.

Permissions & gotchas

  • NSAudioCaptureUsageDescription must be added to Info.plist manually — it's not in Xcode's dropdown. If missing, the stream is created silently dead (no error).
  • System-audio recording is its own TCC permission ("System Audio Recording Only", macOS 14.2+) — separate from Microphone and Screen Recording. One-time prompt; no public API to preflight it.
  • During development, the Info.plist key must be in the parent process (your terminal/IDE), not just the packaged app.
  • Mic capture needs the standard NSMicrophoneUsageDescription + microphone permission.
Spike this first: one GitHub issue (pasrom/meeting-transcriber#79) reports Core Audio process taps failing to capture Microsoft Teams audio specifically. Since Teams is a primary use case, day-one task is a 30-line proof-of-concept: record a Teams call via Electron 39 loopback and confirm audio lands. If per-process taps fail, a system-wide tap or ScreenCaptureKit path is the fallback.
Version caveat: the Electron 39 behavior was documented against v39.0.0-beta.4. Verify against the stable Electron release at build time; keep the Granola-style native Swift helper as plan B (it's ~200 lines with AudioCap as reference).

Design direction (from Mobbin reference sweep)

  • North star: Notion AI meetings / Granola — notes-first. You type during the meeting; recording is a quiet inline element (subtle waveform, Pause/Stop), never a big red dashboard.
  • Detail view: Fireflies' split — editable summary/notes left, timestamped speaker-labeled transcript rail right, playback bar pinned to the bottom.
  • Library: Otter's structure, not its skin — folder sidebar, meeting list with date/duration, but rendered in native-macOS language: vibrancy sidebar, SF Pro, restrained accent color, generous whitespace.
